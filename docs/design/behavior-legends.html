keymap-app · design proposal · not yet implemented

Behavior legends: showing what a key does beyond tap

Today a keycap shows three tap states — primary, shifted, AltGr — in its corners. ZMK keys also hold, double-tap, toggle, switch layers, and run macros. This proposal adds those to the same cap without inventing a second visual language: every mockup below uses the app's existing colors, fonts, and geometry.

The system

Position says when, mark says what

Two orthogonal rules cover every scenario. Where a legend sits on the cap encodes the trigger (tap, hold, double-tap). How the glyph is marked encodes the action type (plain key, layer move, toggle, macro). New behaviors compose from these two axes instead of each getting a bespoke icon.

Trigger — positionAction — mark
tapexisting corners: primary, shifted, AltGr plain keyunmarked glyph, legend gray
holdtop-right slot, solid underline (weight “under” the key) layer moveglyph tinted in the target layer's color — same language as the layer corner tick
double-taptop-right stack; dots = tap count (·· two taps, ··· three), one row per count toggle / latchsmall hollow ring after the glyph: ⇧◦ — “stays on until pressed again”
physicalhoming bar on the bottom edge, bevel-colored — not a legend macroglyph in a dashed chip — “one unit that expands to a sequence”

The top-right corner is free by design — Keycap.tsx already reserves it “for a future layer reference”. The in-cap layer LED dot of the first draft is gone: the layer color now paints the top-right border corner arc as a tick, so the whole interior corner belongs to the behavior stack. Physical features (homing bar) use the bevel steel tones; logical actions use legend white plus layer colors. The two vocabularies never share a color.

Scenario 1 · static

Homing marker

A homing key is a property of the physical board, not of any layer — so it renders on every layer and lives outside layers in the document. Visually it borrows from real keycaps: a short bar on the bottom edge, drawn in the same steel tone as the bottom bevel. Quiet on purpose; it's tactile information, not a binding. 

// document level — new sibling of "layers"
"board": { "homing": ["L-r2-c4", "R-r2-c1"] }
Recommendation Bar, not dot: an in-cap dot could read as a legend or as the toggle ring. Editor: a “Homing key” checkbox in the key editor, labeled as board-wide so it's clear it isn't per-layer.

Scenario 2 · static cap + dynamic detail

Hold-tap, mod-tap, layer-tap

All three are one ZMK concept — a key with a tap action and a hold action (&mt, &lt, custom zmk,behavior-hold-tap). So they get one representation: a hold slot at top-right with a solid underline. When the hold action is a layer move, the glyph is the layer name tinted in that layer's color — the color the user already knows from the layer tabs and LED dots.

Where does Shift+hold = Ä go? Not on the cap. The cap shows one hold glyph (ä); the full matrix lives in the key editor and in a hover tooltip — the dynamic layer of this proposal:

key L-r2-c1 · tooltip / editor detail
tapa ⇧ + tapA holdä ⇧ + holdÄ 
// hold prints a character (mod-tap-style hold-tap)
"L-r2-c1": {
  "primary": "a", "shifted": "A",
  "hold": { "glyph": "ä", "shifted": "Ä" }
}

// hold moves to a layer (layer-tap) — glyph and tint derived, always in sync
"L-r4-c4": {
  "primary": "␣",
  "hold": { "layer": "Nav" }
}
Recommendation Layer-hold legends become clickable: clicking “Nav” on the cap switches the canvas to the Nav layer. That single interaction makes the whole layer graph navigable from the board itself.

Scenario 3 · static

Tap dance — and what “Caps” actually is

A tap-dance binds different actions to tap counts. On the cap: each tap count gets its own row in the top-right stack, prefixed by its count in dots — ·· means two taps, ··· three. The glyph after the dots is what that count produces. A trailing hollow ring ◦ is the action mark for “toggles / stays latched”, not a tap count: solid dots always sit left of the glyph, the ring always right.

To your question — “double-tap Shift → CAPS” has three possible firmware meanings, and the vocabulary distinguishes all three without new machinery: 

// double-tap Shift → Caps Lock (dedicated keycode &kp CAPS)
"L-r3-c0": {
  "primary": "⇧",
  "taps": [ { "count": 2, "glyph": "⇪" } ]
}

// variant: shift-lock via key toggle (&kt LSHFT) — toggle ring on the same glyph
  "taps": [ { "count": 2, "glyph": "⇧", "toggle": true } ]
Recommendation Caps Lock is a dedicated keycode in ZMK (&kp CAPS) — no toggle mark needed, ⇪ already means “locked shift”. The hollow-ring toggle mark exists for the non-obvious cases: &kt LSHFT, &kt LALT (alt-tab without holding), sticky keys later. For actual typing comfort consider &caps_word (auto-off at word end) — it renders as ⇪◦ with the auto-off note in the tooltip. Firmware choice is yours; the visualization covers all three.

Scenario 4 · static cap + dynamic detail

Macros

A macro is not a new slot — it's a kind of binding that any slot can carry (usually primary: “J on the Mouse layer runs Ctrl-C”). On the cap, the macro's display glyph sits in a dashed chip: a border that says “one unit, expands to a sequence”. The sequence itself is a tooltip / editor concern, like the hold matrix above.

Macros are defined once at document level and referenced by name — your mirrored-clipboard example reuses copy/paste across several keys and layers, and one edit updates every cap. 

// document level — macro registry
"macros": {
  "copy":  { "glyph": "⌃C", "label": "Copy",  "steps": "hold Ctrl · tap C" },
  "paste": { "glyph": "⌃V", "label": "Paste", "steps": "hold Ctrl · tap V" }
}

// on the Mouse layer, J is the copy macro
"R-r2-c1": { "macro": "copy" }

Data model

Schema v2 in one look

Everything above lands as schemaVersion: 2. Existing v1 documents stay valid: plain-string slots keep working, the parser upgrades on load. New per-key fields are hold, taps, macro; new document fields are board and macros. The app stays a drawing tool — glyphs are freeform text as today; the structure exists so rendering can mark and tint them, not to compile firmware. 

{
  "schemaVersion": 2,
  "board":  { "homing": ["L-r2-c4", "R-r2-c1"] },
  "macros": { "copy": { "glyph": "⌃C", "label": "Copy", "steps": "hold Ctrl · tap C" } },
  "layers": [ {
    "name": "Base", "color": "#00e5ff",
    "keys": {
      "L-r2-c1": { "primary": "a", "hold": { "glyph": "ä", "shifted": "Ä" } },
      "L-r4-c4": { "primary": "␣", "hold": { "layer": "Nav" } },
      "L-r3-c0": { "primary": "⇧", "taps": [ { "count": 2, "glyph": "⇪" } ] },
      "R-r2-c1": { "macro": "copy" }
    }
  } ]
}

Key editor additions

  • “Homing key” checkbox (board-wide, shown with a board icon so it reads as cross-layer).
  • “On hold” group: glyph field or layer picker or macro picker, plus a toggle checkbox — one shared binding editor.
  • “Tap dance” list: rows of tap-count → same binding editor.
  • Macro manager (small modal or panel section): name, glyph, label, steps.

Open decisions

Choices you should ratify before implementation

1 · Where the hold legend lives

  • PickTop-right slot — the corner the code already reserves; pairs with the layer corner tick; never fights the wide primary legend.
  • Bottom-center (keymap-drawer convention) — collides with the homing bar and squeezes between primary and AltGr on 54 px caps.

2 · Shift+hold (Ä) on the cap?

  • PickTooltip + editor matrix only — the pair is almost always derivable (ä→Ä); five glyphs on a 54 px cap is the readability cliff.
  • Render both (“ä Ä”) in the hold slot — only worth it if you have non-derivable pairs.

3 · Layer-hold glyph source

  • PickDerived from the layer (name + color) — renames and recolors propagate; zero drift.
  • Freeform glyph with manual color — more control, silently goes stale.

4 · Macro identity

  • PickDocument-level registry, keys reference by name — mirrored-clipboard keys stay in sync, tooltips get label + steps for free.
  • Inline per-key macro glyphs — simpler schema, duplicates drift apart.

5 · Layer indicator on the cap

  • PickCorner tick on the border — layer color painted over the top-right corner arc (~7 px arms, bevel width); zero interior space, the behavior stack gets the full corner.
  • In-cap LED dot (first draft) — costs exactly the pixels the hold / tap-dance stack needs.

6 · Dynamic layer scope (phase 2)

  • PickHover tooltip with the full state matrix + click-to-jump on layer-tinted legends — the cap stays a clean static poster (and exports as one); interactivity adds depth without adding ink.
  • Everything on the cap, no dynamics — exports identical, but caps get noisy fast.
